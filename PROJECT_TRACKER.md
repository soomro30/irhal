# Irhal Portal V4 Feature Tracker

Last updated: 2026-06-02

This tracker reflects the current local implementation against the Irhal Enterprise City Guide V4 requirements. Status values are `Not Started`, `In Progress`, `Completed`, or `Blocked`.

## Current Build Snapshot

- Local app: Next.js 16 App Router, TypeScript, Tailwind CSS, and owned shadcn-style UI primitives.
- Arabic is now the default public entry language: `/` renders the Arabic homepage, `/ar` remains available for Arabic locale routes, English public pages are served under `/en`, old unprefixed English public routes such as `/city/karachi` redirect to `/ar/city/karachi`, the header language toggle handles `/`, `/ar`, `/en`, and nested EN/AR routes cleanly, and SEO alternates now use Arabic as `x-default`.
- Public portal UI now uses a custom Irhal travel identity on top of shadcn-style `Button`, `Badge`, and `Card` primitives rather than a generic SaaS/admin look.
- The public header has been redesigned into a modern Lonely-Planet-style clean white sticky bar with backdrop blur, replacing the solid deep-red navigation. Brand identity is preserved through a compact red logo chip (white Irhal PNG on an `irhal-red` rounded badge) and red accents/active states. The desktop nav uses dark-ink links with animated red underline hovers, a Payload-backed City Guides Radix/shadcn DropdownMenu with map-pin city rows, a News link, a real EN/العربية language toggle pill, and an `Explore {city}` red CTA. A new shadcn `DropdownMenu`-based mobile hamburger menu adds previously-missing mobile navigation (Home, News, city guides, Explore CTA, language toggle). All header logic lives in the new client `SiteHeader` component and remains fully RTL-mirrored. The old `city-guide-menu.tsx` was folded into `SiteHeader` and removed. The language toggle now strips existing locale prefixes and rebuilds the opposite canonical locale URL, so `/ar/city/karachi` switches to `/en/city/karachi` instead of falling through the unprefixed Arabic redirect path. A root client locale synchronizer immediately updates `<html lang>`/`dir` during client-side locale navigation so stale RTL/LTR styles cannot persist while the next page streams.
- Arabic header city navigation now localizes Payload city rows for the Arabic UI (Karachi → `كراتشي`, Pakistan → `باكستان`) and renders the desktop and mobile city guide menus with RTL direction/alignment. Frontend route transitions now show immediate feedback through a global top progress indicator plus a public `loading.tsx` skeleton so dynamic server-rendered routes do not appear idle while waiting for the server response. Vercel runtime regions are now pinned in `vercel.json` to `bom1` for nearby South Asia/Middle East execution after Vercel rejected `dxb1` as currently unavailable.
- The item/content detail page (`GuideItemDetail`, used for place/hotel/restaurant/masjid/shopping/tour/family/festival pages) was rebuilt to an enterprise CMS/travel-site standard. It now has: a refined hero (eyebrow, title, lede, scannable Area/Type fact chips, rounded hero image, red "Open map" CTA); a two-column body with the verbatim "Overview" content + a "Good to know" card on the left and a sticky sidebar on the right containing a curated "Quick facts" panel (Area, Type, Price as `$`-symbols, Address + map button — no raw column dump), a "More to explore" cross-category list (~6 items with clear type labels), and a "Plan your trip" CTA; and cross-discovery below the fold — a "You might also like" region with same-category-first rails ("More places/restaurants/hotels in {city}") plus a "Related guides" grid of articles from the same section. Added proper per-item `schema.org` JSON-LD (`guideItemJsonLd` in `seo.ts`, mapping kind → TouristAttraction/Hotel/Restaurant/PlaceOfWorship/ShoppingCenter/Festival/TouristTrip) alongside the breadcrumb JSON-LD. Shared kind-label maps (`guideKindOrder`, `kindPlural`, `kindSingular`) were centralized in `guide-items.ts` and reused by both the detail and section pages. Fully EN/AR + RTL aware.
- City guide section pages (`section/[section]`) were redesigned to match the city-guide template and now use a two-column `main + sidebar` layout instead of an isolated single column. Both section types share it: item sections render the guide-item grid and article sections render article cards, each led by the red "CITY GUIDE SECTION" eyebrow + bold heading. A new reusable `SectionSidebar` component provides an "Explore the guide" section nav (current section highlighted), a "More to explore" cross-discovery card, and a dark "Plan your trip" CTA (View itineraries / Open city map); it is sticky on desktop, stacks on mobile, and is EN/AR + RTL aware.
- Section pages were hardened for production: (1) confusing bare price-tier labels (e.g. "EXPENSIVE") are converted to clear price symbols (`$`–`$$$$`) in guide card meta, and the sidebar "More to explore" items now show an unambiguous type label (Hotel, Restaurant, Shopping, Place, …) instead of the raw category; (2) item sections are now paginated at 24 per page via a `?page=` search param, with a "Showing X–Y of Z" header and an accessible Previous / numbered / Next control (windowed with ellipses), threaded through the Arabic route too; (3) the redundant "Guides and visitor notes / Read deeper visitor notes" block was removed from item sections — after the item list the page goes straight to "Keep exploring {city}". Article-only sections still render their article cards (now without the duplicate heading). Verified in server-rendered HTML: "Showing 1–24 of 110 places" (p1) / "Showing 25–48 of 110 places" (p2), hotel cards rendering `$$$`, sidebar type labels, and the removed notes block.
- Engagement/cross-linking modules were added to every section page to drive more pageviews (Expedia/Lonely-Planet style). The sidebar "More to explore" shows 5–6 items pulled from OTHER categories of the same city (thumbnail + title + category, each linking to its detail page). Below the main content + sidebar, a "Keep exploring {city}" region renders up to three horizontal cross-section item rails (e.g. on Places: More restaurants / More hotels / More shopping), each with an "Explore all" link to that section. Related content is derived once via `getGuideItems`, excludes the current section's kind, is localized, and reuses the existing `GuideItemRail`/card components.
- The breadcrumb bar (in `page-shell.tsx`) was demoted from a second white bar into a quiet soft-gray context strip (`bg-paper-deep`) so the white sticky header reads as the single primary chrome and the breadcrumb is clearly secondary. The current-page item is now a white chip with a subtle border/shadow so it pops against the gray, link items use lighter weight with a `hover:bg-white` state, and the bar is slightly shorter. This produces a clean three-layer depth model (white header → gray breadcrumb → white content) and stays RTL-aware.

Missing item/city media renders the clean per-kind SVG placeholder; CMS records stay photo-empty until an editor attaches proper R2 media in Payload.
